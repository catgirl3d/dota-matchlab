# Словарь описаний талантов Dota 2

## Назначение

В детальной аналитике матча STRATZ передаёт факт изучения таланта и его числовой `abilityId`, но не готовую строку описания. Поэтому UI использует статический словарь:

```text
abilityId -> готовое описание таланта
```

Сгенерированный файл:

```text
src/react-app/lib/talent-descriptions.ts
```

Пример записи:

```ts
6900: "+12 Desolate Damage",
```

Файл является производным артефактом. Его не следует редактировать вручную: при следующем обновлении Dota 2 изменения будут перезаписаны генератором.

## Запуск генерации

Из корня проекта:

```bash
pnpm generate:talent-descriptions
```

Генератор заново скачивает текущие данные и записывает словарь. После генерации рекомендуется выполнить:

```bash
pnpm test
pnpm typecheck
pnpm lint
pnpm build
```

Если в worktree отсутствует `node_modules`, сначала установите зависимости:

```bash
pnpm install --frozen-lockfile
```

## Источники данных

Генератор находится в:

```text
scripts/generate-talent-descriptions.mjs
```

Он использует несколько источников с разными обязанностями.

### Официальный datafeed Dota 2

Список героев:

```text
https://www.dota2.com/datafeed/herolist?language=english
```

Для каждого героя запрашивается:

```text
https://www.dota2.com/datafeed/herodata?language=english&hero_id=<ID>
```

Из ответа берутся:

- текущие таланты из `heroes[].talents`;
- числовой `talent.id`;
- актуальный шаблон отображения из `talent.name_loc`;
- дополнительная информация из `special_values[].bonuses`, если она присутствует в API.

Проверка по `heroes[].talents` важна: устаревший `dotaconstants` может содержать старые таланты, которые уже удалены из текущей версии игры. Такие записи в итоговый словарь не попадают.

### Распакованные Valve KeyValues-файлы

Современный клиент Dota 2 использует не только старый общий файл `npc_abilities.txt`, но и split-файлы героев:

```text
scripts/npc/heroes/npc_dota_hero_<hero>.txt
```

Актуальный репозиторий с распакованными файлами:

```text
https://github.com/spirit-bear-productions/dota_vpk_updates
```

Генератор скачивает файлы из:

```text
https://raw.githubusercontent.com/spirit-bear-productions/dota_vpk_updates/main/scripts/npc/heroes/
```

Внутри KeyValues значения талантов находятся в `AbilityValues`. Возможны два варианта.

Талант задан непосредственно:

```text
"special_bonus_unique_beastmaster_5"
{
    "AbilityValues"
    {
        "bonus_ms" "15"
    }
}
```

Или талант является модификатором значения обычной способности:

```text
"pull_strength"
{
    "value" "200"
    "special_bonus_unique_naga_siren_reel_in_speed" "+125"
}
```

Генератор поддерживает оба варианта.

### dotaconstants

Если локальные файлы отсутствуют, генератор скачивает:

```text
https://raw.githubusercontent.com/odota/dotaconstants/master/build/abilities.json
https://raw.githubusercontent.com/odota/dotaconstants/master/build/ability_ids.json
```

Если в корне проекта присутствуют локальные `abilities.json` и `ability_ids.json`, они используются прежде всего как fallback-метаданные. Для текущих талантов приоритет имеют ID и `name_loc` из официального datafeed.

## Как работает алгоритм

1. Читаются локальные `abilities.json` и `ability_ids.json`; при отсутствии они загружаются из `dotaconstants`.
2. Из официального `herolist` строятся список numeric hero ID и имена файлов `npc_dota_hero_*.txt`.
3. Для всех героев параллельно, с ограничением в 8 запросов, загружаются datafeed-ответы и KeyValues-файлы.
4. Из каждого datafeed-ответа извлекаются только таланты, которые существуют в текущем списке героя.
5. Из `name_loc` берётся актуальный шаблон, например:

   ```text
   +{s:bonus_bonus_damage} Desolate Damage
   ```

6. KeyValues преобразуется в дерево `Map` и из `AbilityValues` собираются связи:

   ```text
   имя таланта -> имя special value -> число
   ```

7. Числа из KeyValues имеют приоритет над fallback-значениями datafeed.
8. Плейсхолдеры `{s:...}` заменяются числовыми значениями.
9. Если значение отрицательное, а знак уже записан в шаблоне, повторный знак удаляется. Например `-4`, а не `--4`.
10. Если в шаблоне остался неразрешённый плейсхолдер, генерация завершается ошибкой и файл не считается валидным.
11. Результат сортируется по numeric ID и записывается в `talent-descriptions.ts`.

## Особые случаи

### Стандартные таланты

Для стандартных записей значение часто зашито в имени:

```text
special_bonus_attack_range_75
```

Если шаблон использует `{s:value}`, генератор извлекает `75` из имени таланта.

### Несколько вариантов одной способности

Официальный datafeed иногда возвращает legacy или alternate-способность с тем же talent ID. Для таких конфликтов используется первое значение в порядке основного ответа. KeyValues-значение из актуального файла имеет приоритет.

### Удалённые и устаревшие таланты

Если талант есть в старом `abilities.json`, но отсутствует в `herodata[].talents`, он пропускается. Это предотвращает попадание в словарь удалённых талантов.

## Проверка результата

После запуска проверьте:

```text
src/react-app/lib/talent-descriptions.ts
```

Файл должен начинаться так:

```ts
export const talentDescriptions: Record<number, string> = {
```

Для ручной проверки Spectre:

```text
6900: "+12 Desolate Damage",
```

В сгенерированном файле не должно быть:

- строк с `{s:`;
- записей `TALENT` вместо описания;
- двойных знаков вроде `--4s`;
- шаблонов без числового значения.

Количество записей не является фиксированным: оно может измениться после патча, добавления героя или удаления таланта.

## Обновление после патча Dota 2

Рекомендуемый порядок:

1. Дождаться, пока официальный datafeed и репозиторий с распакованными файлами обновятся.
2. Выполнить `pnpm generate:talent-descriptions`.
3. Проверить количество и несколько изменившихся записей в `src/react-app/lib/talent-descriptions.ts`.
4. Запустить `pnpm test`.
5. Запустить `pnpm typecheck`, `pnpm lint` и `pnpm build`.
6. Просмотреть diff сгенерированного файла и включить его в коммит вместе с изменениями генератора, если они понадобились.

Автоматического расписания обновления нет: словарь обновляется вручную через команду генерации, чтобы изменение данных Dota 2 происходило явно и попадало в review.

## Диагностика ошибок

### `Could not fetch ...`

Недоступен один из удалённых источников, изменился URL или репозиторий ещё не синхронизирован после патча. Повторите запуск позже и проверьте URL из сообщения об ошибке.

### `Could not resolve ... talent description(s)`

Для текущего таланта не найдено числовое значение. Проверьте:

1. Есть ли talent в официальном `herodata`.
2. Есть ли соответствующий `special_bonus_*` в hero KeyValues-файле.
3. Не изменился ли формат `AbilityValues`.
4. Не появился ли новый тип placeholder, который не учитывает `resolveTalentDescription`.

Не следует просто удалять такую запись из списка ошибок: это скроет неполное описание. Нужно добавить поддержку нового формата и тест на минимальном KeyValues-примере.

### Изменился путь файлов Valve

Обновите константу `DOTA_VPK_HERO_FILE_URL` в `scripts/generate-talent-descriptions.mjs`. Если Valve снова объединит или иначе разделит файлы, потребуется изменить функцию `getHeroAbilityFileNames` или сборщик `getTalentBonusesFromKeyValues`.

## Тесты генератора

Unit-тесты находятся в:

```text
scripts/generate-talent-descriptions.test.mjs
```

Они проверяют:

- подстановку числовых значений;
- стандартные таланты с `{s:value}`;
- прямые и связанные KeyValues-значения;
- актуальные шаблоны вместо устаревших локальных;
- удаление двойного знака;
- отказ при неразрешённом placeholder;
- игнорирование legacy-талантов.

Запустить только эти тесты:

```bash
pnpm test:talent-descriptions
```
